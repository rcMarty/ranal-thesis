\section{Základní funkce a model spolupráce}
\label{sec:github:zakladni_funkce_model_spoluprace}
V~této sekci jsou popsány nejpoužívanější funkce GitHubu a~modely interakce mezi vývojáři~\cite{github_docs}.

\subsection{Repozitáře}
\label{subsec:github:repozitare}
Repozitář představuje základní prvek na GitHubu. Jedná se o~místo, kde je ukládán zdrojový kód a~jiné soubory projektu. Repozitáře mohou být veřejné nebo soukromé a~mohou mít více spolupracovníků~\cite{github_docs}.

\subsubsection*{Forky}
\label{subsubsec:github:forky}
Fork je repozitář vytvořený pod jiným uživatelským účtem, který vychází z~původního repozitáře a~sdílí jeho historii. Forky jsou využívány primárně pro přispívání do cizích projektů. Nejčastěji je vytvořen fork, v~něm jsou provedeny změny a~následně je otevřen \PR do původního repozitáře. Možnost navrhnout sloučení změn do původního projektu je hlavním rozdílem mezi forkem a~klonem repozitáře. Fork se tedy částečně chová jako samostatná větev, avšak bez nutnosti mít přímý zápis do původního repozitáře. Tento způsob práce umožňuje bezpečně experimentovat bez přímého ovlivnění původního projektu~\cite{github_pull_requests_docs}.

Každý fork je plně samostatný a~má vlastní větve, členy, issues, pull requesty, wiki a~další funkce~\cite{github_docs}.

\subsection{Issues}
\label{subsec:github:issues}
Issues (neboli problémy) představují jednu ze základních a~nejvyužívanějších funkcí GitHubu. Jsou používány ke sledování chyb, nápadů, požadavků na funkce a~dalších úkolů spojených s~projektem~\cite{github_issues_docs}.

Jedna issue může mít sub-issues, aby mohl být větší úkol rozdělen na více menších částí. Pokud je issue závislá na dokončení jiné issue, lze tuto závislost explicitně vyjádřit a~označit úkol jako blokovaný. Současně mohou být přidávána metadata, například labely, typ issue, přiřazení nebo milníky. Issue může být přiřazena konkrétnímu uživateli, což usnadňuje organizaci práce i~sledování odpovědnosti~\cite{github_issues_docs}.

\subsection{Pull Requests}
\label{subsec:github:pull_requests}
Pull requests (\PR) představují mechanismus, kterým může být navrženo začlenění změn do projektu. Jde o speciální případ Issue takže všechny pr jsou i issues ale ne všechny issues jsou PR. Umožňují vytvořit požadavek na provedení změn ve zdrojovém kódu. Následně může proběhnout kontrola změn a~diskuze nad konkrétními částmi kódu. Tím je podporováno udržení kvality kódu a~včasné odhalení chyb~\cite{github_pull_requests_docs}.

Za zmínku stojí také dobrá provázanost mezi issues a~\PR. Pokud \PR řeší konkrétní issue, je možné tuto issue automaticky uzavřít pomocí klíčových slov, například \code{Closes \#123}, kde \code{\#123} představuje číslo issue~\cite{github_pull_requests_docs}.

Typický workflow zahrnuje vytvoření forku a~založení nové větve, například pro konkrétní issue. Poté jsou provedeny změny, které jsou odeslány do forku, a~následně je z~dané větve otevřen \PR{} směrem do původního repozitáře. Pokud změny jsou schváleny, může být \PR{} sloučen~\cite{github_pull_requests_docs,psl_git_tutorial_main}.

\subsection{Labels}
\label{subsec:github:labels}
Labely představují klíčový nástroj pro kategorizaci a~prioritizaci nejen issues, ale také \PR a~diskuzí. Labely jsou vytvářeny v~rámci repozitáře a~následně přiřazovány jednotlivým issues, pull requestům a~diskuzím. V~každém novém repozitáři je k~dispozici základní sada labelů, například \code{bug}, \code{enhancement}, \code{question} nebo \code{help wanted}. Tyto labely mohou být podle potřeby upravovány, přidávány i~odstraňovány~\cite{github_issues_docs}.

\subsection{GitHub Actions}
\label{subsec:github:actions}
Platforma GitHub kromě hostování repozitářů poskytuje také řešení pro automatizaci procesů, jako je \CICD{}. Toto řešení se nazývá GitHub Actions a~umožňuje přímo v~prostředí GitHubu vytvářet a~spouštět vlastní workflow~\cite{github_actions_docs}.

\subsubsection*{Triggery}
\label{subsubsec:github:actions_triggery}
Triggery jsou události, které spouštějí workflow. Mezi nejběžnější patří \code{push}, \code{pull\_request}, \code{schedule} pro plánované spouštění a~\code{workflow\_dispatch} pro manuální spuštění. Triggery umožňují flexibilně reagovat na různé události v~repozitáři~\cite{github_actions_docs}.

\subsubsection*{Workflow}
\label{subsubsec:github:actions_workflow}
Workflow je definováno v~souboru s~příponou \code{.yml} nebo \code{.yaml}, který je umístěn v~adresáři \code{.github/workflows/}. Jedná se o~automatizovaný proces složený z~definic jednotlivých kroků. V~repozitáři může existovat více workflow, které se spouštějí na základě různých podmínek, například pouze při \code{push} do hlavní větve~\cite{github_actions_docs}.

\subsubsection*{Jobs a Steps}
\label{subsubsec:github:actions_jobs_steps}
Workflow se skládá z~jednotlivých úloh, které se označují jako jobs. Každý job je dále tvořen dílčími kroky, které se označují jako steps. Jobs standardně běží paralelně, lze je však nakonfigurovat tak, aby na sebe navazovaly. Každý step může představovat buď příkaz spuštěný v~shellu, nebo použití předdefinované akce (action). Steps oproti Jobs běží sekvenčně~\cite{github_actions_docs}.

\subsubsection*{Runners}
\label{subsubsec:github:actions_runners}
Runners zajišťují běh jednotlivých workflow a~jejich kroků. Jedná se o~virtuální~servery, které mohou být hostovány buď GitHubem, nebo si je může uživatel provozovat sám~\cite{github_actions_docs}.

Jednou z~výhod GitHub Actions je integrace s~\PR. Při vytvoření nového \PR mohou být automaticky spuštěny workflows a~jejich výsledek je následně přehledně zobrazen přímo u~daného \PR. Takové workflows často obsahují testy, kontrolu sestavení nebo ověření pravidel formátování kódu. Tím je zajištěna další vrstva kontroly kvality a~je sníženo riziko, že se do hlavní větve dostane změna narušující funkčnost projektu~\cite{github_actions_docs,github_pull_requests_docs}.
